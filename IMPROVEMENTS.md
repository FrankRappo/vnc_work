# vnc_work — что докрутить для скорости и надёжности

Заметки по итогам первых боевых VNC/CDP-сессий на реальных кабинетах и формах.
Главный вывод: **большая часть времени уходила не на задачу, а на (1) угадывание координат по скринам
и (2) накладные расходы на отдельный chromium-connect на каждое действие.** Ниже — как это убрать.

## P0 — самые большие ускорения

### 1. Перенести ВСЁ взаимодействие на CDP (сделано в движке — закрепить дисциплиной)
На сессии ~половина шагов — это `xdotool` по координатам с двойными перепроверками скрином, потому что
Read рендерит jpg в разном масштабе. Через `click '<text>'` / `clicksel` / `type '<sel>'` это один точный
шаг без скрина-сверки. **Правило: координаты (`xclick`) — только для не-DOM (chrome-попап, canvas, капча).**
Эффект: −50% шагов, −почти все «промахи».

### 2. Долгоживущая CDP-сессия вместо `node cdp.mjs` на каждую команду
Сейчас каждая команда = новый процесс node + `puppeteer.connect` + disconnect (сотни мс + спавн).
Сделать `cdpd.mjs` — демон: один раз `connect`, читает команды из FIFO/сокета (`/tmp/vnc_work.cmd`),
пишет ответы. `vnc.sh` шлёт строки в FIFO. Экономит ~0.5–1 c на каждое действие, а их десятки.

### 3. Flow-файлы (батч шагов в одном проходе)
Дать `vnc.sh flow steps.json`, где steps = `[{do:"type",sel:"…",val:"…"},{do:"click",text:"LOG IN"},{do:"wait",sel:"…"},{do:"shot",label:"…"}]`.
Один node-процесс прогоняет весь сценарий, скринит по ходу. Логин в ЛК = 1 вызов вместо 6. И это сразу
**runbook** (см. P2.12) — повторный вход воспроизводится одной командой.

### 4. Перехват сетевых ответов (CDP Network) — данные часто НЕ в DOM
Многое (токены, список ПВЗ виджета, ответы API кабинета) приходит в XHR/fetch, а не в верстку.
Команда `net '<url-regex>' [last N]` — включить `Network.responseReceived`/`getResponseBody` и отдать тело.
На этой сессии так можно было сразу вынуть `account_id`/конфиг, не кликая по профилю.

### 5. `shoot` = scrot+jpg+печать пути одним вызовом + опциональный авто-скрин после каждого действия
Уже почти так; добавить флаг `VNC_AUTOSHOT=1`, чтобы `click/type/go` сами делали скрин — меньше ручных шагов.

## P1 — надёжность и неудобные места

### 6. Сетка-оверлей для редких координатных кликов (canvas/капча)
Когда без `xclick` никак (капча-картинка, точка на карте): хелпер `gridshot` рисует на скрине сетку 100px
с подписями координат (ImageMagick `-draw`) → модель целится по нативным пикселям точно, без пересчёта масштаба.

### 7. Авто-дисмисс типовых баннеров
Профильные prefs уже глушат save-password/гео/нотификации. Добавить набор правил для cookie-баннеров и
«Save password»-остатков: `dismiss` пробегает по списку текстов («Принять», «No thanks», «Понятно», ×) и кликает.

### 8. `wait-text '<regex>'` и `assert-url <regex>`
Ждать не только селектор, но и появление текста; проверять, что переход случился. Меньше «слепых» sleep.

### 9. Коды: почта — есть; SMS — канал к человеку
`fetch_email_code.py` готов; добавить `--wait` (поллинг новых писем N сек — код приходит с задержкой).
Для SMS автоприёма нет (нет шлюза) → сделать аккуратный хендофф: пинг человеку в TG «нужен код для X» +
ожидание ответа (через тот же tg-бот, что у проекта). Капча — TigerVNC-хендофф (как сейчас).

### 10. Запасной ввод для «капризных» React-инпутов
Если `type` (CDP) не «прилипает» к контролируемому полю — фолбэк на `bbox`→`xclick`→`paste`. Обернуть в
`type` авто-фолбэком (проверить value после ввода, при пустом — повторить через clipboard).

## P2 — удобство/масштаб

### 11. Мульти-таргет
Поддержать несколько вкладок/окон: `tabs`, `tab <n>`, чтобы вести OAuth-redirect в одной, кабинет в другой.

### 12. Запись/реплей runbook'ов
Логировать каждое действие в flow-файл `runbooks/<task>.json`. Вместе с персистентным профилем это даёт
идемпотентный повторный вход/настройку «одной кнопкой» (профиль протух → реплей логина).

### 13. Профили под задачи + бэкап сессии
`VNC_PROFILE` per-задача (billing/support/admin и т.п.). `backup.sh`/`restore.sh` (как в chatgpt_integration) — снапшот
cookies, чтобы не проходить капчу заново после переустановки/чистки.

### 14. Интеграция с оркестратором
Шаблон таска «VNC-веб-задача»: преамбула + ссылка на этот README + точка хендоффа человеку (TG-пинг при
SMS/капче). Тогда часть веб-онбордингов можно отдавать саб-агенту, а не вести вручную.

## Чего НЕ автоматизировать
- **SMS-код, подпись через ЭДО, юридические согласия, привязка банковского счёта** — это действия владельца
  (риск + ответственность). Инструмент доводит до этого шага и зовёт человека, дальше — он.
- Капчу решаем сами по скрину, **но** при залипшей reCAPTCHA — хендофф, не долбить (бан IP).

## Метрика «стало быстрее»
Цель: типовой «вход в ЛК + достать ключ» — **≤8 действий без единого координатного клика и без сверки-скрина
на каждый шаг** (сейчас было ~25 с перепроверками). Реализация P0.1–P0.3 закрывает основную часть.
